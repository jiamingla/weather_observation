# Stage 2 — 指標計算 + 可信度旗標

在 Stage 1 的基礎上加入「近 5 日 / 近 20 日平均最高溫」逐日序列，並擴充可信度旗標告訴前端「什麼時候不要顯示」。

## 為什麼把指標放在後端

三個理由：

1. **單一事實來源** — AI 分析層與前端圖表拿到的是**同一份**計算結果，不會出現「圖表顯示 30.78°、AI 引用 30.82°」這種矛盾。
2. **方便對拍驗證** — 純後端的輸出可用 `curl` 直接 snapshot，指標是否正確一目了然；前端算就要打開瀏覽器 DevTools。
3. **純函數易測** — [metrics.ts](src/weather/metrics.ts) 沒碰 I/O，未來要寫單元測試隨時能補（本次刻意不寫）。

## 計算規則

### 近 N 日移動平均（rollingAverage）

位置 `i` 的 MA = `avg(values[i-N+1 .. i])`。

**嚴格規則**：視窗內任一天為 `null`（缺漏），當天 MA 一律輸出 `null`，**不做部分平均**。

理由：題目明確要求「資料筆數不足 / 資料缺漏 → 指標無法計算 → 不應輸出可能誤導的結果」。若一個 5 日窗裡缺了 1 天就用 4 天平均頂替，使用者會以為自己拿到的是 5 日平均，這正是題目想避免的誤導。

### 當資料不足視窗

前 `N-1` 天一律輸出 `null`（例：前 4 天沒有 MA5、前 19 天沒有 MA20）。這段 null 在前端要以「無資料」狀態渲染，不要畫線、不要插值。

### 為什麼保留整條時序而不只回傳「目前值」

題目要求前端顯示**平均線**（不是數字卡片），所以需要逐日 MA 讓前端畫線。但同時也回傳 `latestMa5` / `latestMa20` 讓 AI 分析層與數字卡片有單一值可引用。

## 回應擴充

在 Stage 1 的 `WeatherPayload` 上新增 `metrics` 與兩個 credibility 欄位：

```ts
metrics: {
  ma5MaxTemp: Array<{ date: string; value: number | null }>;    // 長度 = 30
  ma20MaxTemp: Array<{ date: string; value: number | null }>;   // 長度 = 30
  latestMa5: { date: string; value: number } | null;            // 最後一個非 null
  latestMa20: { date: string; value: number } | null;
}

credibility: {
  // ...（Stage 1 原有欄位）
  ma5Computable: boolean;    // latestMa5 !== null
  ma20Computable: boolean;   // latestMa20 !== null
  notes: string[];           // 新增指標相關中文說明，例如：
                             //   "近 5 日平均最高溫無法計算（視窗內有缺漏日）"
                             //   "近 20 日平均最高溫最新可信日為 ... 較區間末日延遲"
}
```

## 手算驗證（對拍臺北 466920）

```
今天：2026-04-14     範圍：2026-03-15 ~ 2026-04-13（30 天）
CWA 最新資料：       2026-04-12（4-13 尚未發布 → 已標記為缺漏）
```

### MA5 @ 2026-04-12

窗：04-08 ~ 04-12，最高溫序列 `[24.5, 30.8, 33.2, 32.1, 33.3]`
手算：`(24.5+30.8+33.2+32.1+33.3) / 5 = 153.9 / 5 = 30.78`
API 回應：`{ date: "2026-04-12", value: 30.78 }` ✅

### MA20 @ 2026-04-12

窗：03-24 ~ 04-12，20 筆最高溫總和 `533.9`
手算：`533.9 / 20 = 26.695` → 進位 `26.7`
API 回應：`{ date: "2026-04-12", value: 26.7 }` ✅

### MA5 @ 2026-04-13（最後一日）

窗：04-09 ~ 04-13，序列 `[30.8, 33.2, 32.1, 33.3, null]`
因含 null → `value: null` ✅（若此處被回傳 `32.35`，就代表規則被破壞）

## 對應題目 (D) 驗證要求

| 題目問題                      | 本階段的答案素材                                                |
| --------------------------- | -------------------------------------------------------------- |
| 任選一天驗證 MA5           | 上面 04-12 的 30.78 對拍通過                                    |
| 確認 MA20 正確              | 上面 04-12 的 26.7 對拍通過                                     |
| 資料不足 20 日如何處理       | 前 19 天一律輸出 `null`；credibility `ma20Computable=false`     |
| 缺漏值如何處理               | 視窗含 null → MA 視為不可計算，不做內插                        |
| 異常值判斷                   | Zod 校驗欄位型別 + schema 僅接受 Number；`"T"` 特殊處理於雨量  |

## 對應題目 (E) 風險處理

| 情境                    | 系統行為                                                      |
| ---------------------- | ------------------------------------------------------------ |
| API 連線失敗            | NestJS 拋 500；前端顯示錯誤並不渲染圖表                       |
| 回傳筆數不足            | `credibility.completeDays` 與 `missingDates` 明確列出       |
| 日期排序錯誤            | Service 以 `enumerateDates(from, to)` 作為骨架，強制遞增順序 |
| 欄位缺失或為空          | Zod 校驗失敗則拋錯；個別日資料缺漏則對應欄位為 `null`       |
| 不同來源資料不一致       | 溫度與雨量分屬兩個 endpoint，以日期 join；任一缺漏即標記   |
