# Stage 4 — 城市 / 地名模糊匹配（白名單 + 字串比對 + AI Fallback）

題目 (A).1 模組 4 原本列為「若時間不足可放棄」。實作完所有核心功能後回頭補上。設計重點：**使用者可以輸入「屈尺」這種村里級地名，系統也能給出可驗證的對應**，同時**絕不讓 AI 回傳白名單外的 ID**。

## 雙入口並存（符合使用者需求）

| 入口 | 特色 | 風險 | 適合情境 |
| --- | --- | --- | --- |
| 快捷鈕（5 個） | 一鍵直查，無 AI | 無 | 台北 / 台中 / 高雄 / 花蓮 / 嘉義 |
| 測站代號輸入 | 使用者自帶 ID | 無 | 已知測站號 |
| **地名輸入（AI 模糊匹配）** | 支援村里、舊稱、別名 | AI 幻覺 → 白名單擋 + 使用者確認 | 「屈尺」「墾丁」「羅東」 |

地名輸入**不會直接換 stationId**，必須使用者點「採用此測站 →」才會啟動查詢。

## 三層匹配策略

```
使用者輸入
  │
  ▼
┌──────────────────┐
│ 1. 字串正規化      │  「台北」→「臺北」，小寫、trim
└────────┬─────────┘
         ▼
┌──────────────────┐
│ 2. 精確匹配        │  比對 name / city / aliases
└────────┬─────────┘  命中 → confidence: high
         │ 沒命中
         ▼
┌──────────────────┐
│ 3. 子字串匹配       │  僅當「唯一」命中才直接回
└────────┬─────────┘  命中 → confidence: medium
         │ 多個命中或沒命中
         ▼
┌──────────────────┐
│ 4. AI Fallback    │  Gemini 2.5-flash-lite
│   + 白名單 enum    │  回傳必須是清單內 ID
└────────┬─────────┘  命中 → confidence: low（需使用者確認）
         │ 白名單外
         ▼
    verdict: 'none'（拒絕）
```

為什麼第 3 步要「唯一才回」：輸入「臺」會子字串匹配到 6 個測站，直接回第一個會誤導；這時交給 AI 按地理距離決定最合適的。

## 白名單（[stations.ts](src/station/stations.ts)）

27 個手動策展的 CWA 署屬有人氣象站，涵蓋全台 22 縣市 + 離島（馬祖 / 澎湖 / 金門 / 蘭嶼）。每個測站有：
- `id` — CWA 官方站號
- `name` — 官方名稱
- `city` — 所在縣市
- `aliases?` — 常見別名（含繁簡互換、英文、鄰近地名）

**刻意不收全台 400+ 自動測站**，理由三點：資料完整度（署屬站 API 支援好）、可驗證性（手動策展我能複核每筆）、地理覆蓋（30 個點足矣）。

## AI Fallback 的 4 道護欄

1. **Temperature: 0** — 降低創造性
2. **responseSchema** — 強制輸出 `{ stationId: string, reason: string }` 結構
3. **Prompt 規則** — 不得編造、不得回清單外的 ID、若無對應必須回空字串
4. **後端白名單驗證** — 收到 AI 回應後再用 `STATIONS.find(id === stationId)` 確認一次；若不在清單內，拒絕並在 message 中明列 AI 理由（透明給使用者看）

## 信心度與 UI 配色

| confidence | 情境 | 前端配色 | 是否自動採用 |
| --- | --- | --- | --- |
| high | 精確匹配 | 綠底 | 使用者按下「採用」才換 |
| medium | 唯一子字串匹配 | 黃底 | 同上 |
| low | AI 推測 | 黃底（警示） | 同上，且 message 必含「請確認是否採用」 |
| none | 找不到 | 紅底 | 無法採用 |

**為什麼不自動採用**：即使 high confidence 也可能出現同名不同縣市的情況（例如「中壢」對應新屋站，而非桃園站）。使用者確認是最後一道防線，也呼應 (E) 「不隱匿決策邏輯」原則。

## API

### `GET /api/station/list`
回傳白名單（id / name / city），供前端顯示所有可選測站。

### `POST /api/station/match`
```json
// request
{ "input": "屈尺" }

// response
{
  "stationId": "466880",
  "stationName": "板橋",
  "city": "新北市",
  "method": "ai",
  "confidence": "low",
  "message": "AI 推測「屈尺」最接近測站「板橋」（新北市）。理由：屈尺位於新北市新店區，距離板橋測站最近。請確認是否採用。"
}
```

## 踩雷紀錄

### Windows bash + curl 中文 body 亂碼

測試時 `curl -d "{\"input\":\"屈尺\"}"` 在 Git Bash 下 JSON body 的中文被編碼成 Big5 才送出，導致 AI 收到 `���B` 亂碼、誤判成「不是地名」。這是**開發工具問題，不是程式 bug**，但如果只靠 curl 測試會誤以為 AI 判讀爛，錯殺整個設計。

**解法**：用 `printf '{"input":"%s"}' "$q" > /tmp/body.json && curl --data-binary @/tmp/body.json` 確保 UTF-8；或直接在瀏覽器前端測。

**教訓**：跨編碼的系統（terminal → HTTP → Node → LLM）要先確認每一段的 encoding，不要把「看起來應該是 UTF-8」當真。
