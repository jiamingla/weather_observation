AI 使用紀錄 — 天氣觀測資料系統
=============================================
工具：Claude Code (Claude Opus 4.6, 1M context) — 用於設計、寫碼、除錯
      Google Gemini 2.5-flash-lite 經 @google/genai SDK — 作為受限版 AI 分析的生成器
涵蓋範圍：範圍取捨決策 → Stage 1~3 (後端資料、指標、前端圖表)
        → Stage 5 (AI 分析受限版) → Stage 4 (地名模糊匹配)


─────────────────────────────────────────────
(C).1 我最初如何定義問題給 AI
─────────────────────────────────────────────

我沒有先讓 AI 寫程式，而是先把題目檔丟給 AI 並說：
「這是工程部高管給我們的作業，我想先決定哪些要做哪些不要做。」

用意是先拿到範圍判斷，不是直接生成系統。我希望 AI 先幫我：
  - 拆解題目評分權重（核心功能 vs AI 分析）
  - 標出可砍項目與為什麼可以砍
  - 就我預定的技術棧（NestJS + Nuxt3 + Postgres + Cron）挑戰是否過度

這樣做的好處是：後續任何 AI 寫出的程式碼，我都可以對照「當初的
範圍取捨計畫」判斷是否符合作答策略，避免 AI 把我帶進多餘功能。

具體 prompt 順序：
  1. 「請看題目檔並給我範圍取捨建議」—> 得到必做 / 限縮做 / 放棄三類
  2. 「看看我寫的 A 部分，給我建議」—> AI 指出我 A1 模組超過 6 個、
      A3 漏掉放棄的項目、A4「AI 會錯在單元測試」太弱，引導我改寫
  3. 決定好範圍才說「現在我想做 後端：資料擷取 + Zod 校驗這一塊」


─────────────────────────────────────────────
(C).2 AI 曾產出哪些不可靠或錯誤的內容？我如何辨識？
─────────────────────────────────────────────

● 錯誤 1：誤判資料來源要用爬蟲

  AI 一開始搜尋「CWA 歷史 30 天觀測 API」得到的結果多半指向 codis
  這個 SPA 頁面，於是它跟我說「沒有官方 REST 端點，要爬 HTML 或
  想辦法從 JS 反查 endpoint」。我差點就接受這個方向。

  我怎麼辨識的：
  - AI 的結論是「搜尋結果沒提到」，不是「我打過 API 確認沒有」
  - 我主動去 https://opendata.cwa.gov.tw/dist/opendata-swagger.html
    自己翻 swagger，貼連結給 AI 並質疑：「這裡看起來有氣象跟雨量的
    過去觀測資料」

  結果：AI 直接用 curl 抓 swagger 原始 JSON（不是 swagger UI 的
  HTML，那是空殼），找到 7099 行 `C-B0024-001 30 天觀測資料-地面
  測站觀測資料`，完全對應題目需求，一行爬蟲都不用寫。

  教訓：當 AI 說「找不到 X」時，要分清楚是「確認不存在」還是
  「搜尋結果的前三項沒提到」。後者我要自己去主來源驗證。


● 錯誤 2：若讓 AI 直接看 swagger 寫 Zod schema，會炸掉

  Swagger 文件上，雨量欄位型別寫 String，最高溫/最低溫寫 Double。
  如果我讓 AI 直接根據文件寫 Zod：
      Precipitation: z.string().transform(parseFloat)
  這段程式上線會馬上爆炸，因為 CWA 的雨量值除了數字字串
  （"0.0"、"3.5"）外，還會回 "T" 表示 trace（微量 <0.1mm）。
  parseFloat("T") = NaN，後面所有計算直接污染。

  我怎麼辨識的：
  - 我堅持「先 curl 真實 API 再寫 schema」，不相信文件
  - 在實際回應裡看到 "Precipitation":"T" 這個值
  - 查證後確認是官方 trace 標記，且 swagger 完全沒寫

  調整：在 cwa.schemas.ts 寫了 parseRainfall(raw) 明確處理三種情境
  （trace / 缺漏 / 正常數字），並在輸出 payload 中保留
  rainfallTrace 旗標供 AI 分析層與 UI 使用。

  教訓：文件與真實世界有落差。凡是要餵給計算的欄位，一定要用
  真實樣本校驗型別假設。


● 錯誤 3：AI 若自動寫 rollingAverage，可能做「部分平均」

  題目明確要求「資料筆數不足 → 指標無法計算 → 不應誤導」。但 AI
  寫的 moving average 若沒提醒，往往會在缺漏日就直接略過 null、
  用剩下的 4 天算平均（看起來很貼心，實際上違反題目可信度紅線）。

  我怎麼辨識的：
  - 在寫 metrics.ts 之前，我先在對話裡寫出規則：「任一 null 就
    輸出 null，不做部分平均」
  - 用 04-13 這個 CWA 尚未發布的日子當測試 case，確認 MA5 @ 04-13
    確實是 null、而非 32.35（= 4 天平均）

  教訓：規則要先寫成文字、再寫成 code，否則 AI 的「預設善意」會
  覆蓋掉題目的明確要求。


● 錯誤 4：NestJS 的 pnpm 安裝步驟在 Windows 上悄悄失敗

  @nestjs/cli 在 Windows 呼叫 pnpm 時拼錯指令（變成 `pnpm pnpm
  install`），跳出看起來很和善的 ASCII art 訊息「Installation
  in progress...」然後失敗，但沒有明顯錯誤行。AI 差點說「已經裝
  好了」就接著寫程式。

  我怎麼辨識的：
  - 我看了 scaffold 指令的輸出最後幾行，抓到
    Failed to execute command: pnpm pnpm install
  - 手動進 backend/ 跑 pnpm install 才真的裝好依賴

  教訓：AI 會讀粗略的「成功」訊息，但不一定會驗證底層副作用。
  涉及安裝、build、deploy 的指令要自己看完整 log。


● 錯誤 5（Stage 5 踩的第一個雷）：AI API key 格式看起來對但實際不對

  我原本以為拿到「任何長字串」就能當 API key，直接貼進 .env。第一次
  設定 GEMINI_API_KEY 是 AQ.Ab8RN6I4... 開頭，呼叫時一直回 403 /
  429，錯誤訊息裡 free_tier_requests, limit: 0 看起來像 quota 問題。

  我怎麼辨識的：
  - 對照 Google AI Studio 文件，發現官方 key 應該以 AIza 開頭，
    AQ. 開頭是 Google Cloud OAuth token，根本不是 AI Studio key
  - 因此即使 API 回 429 而非 401，真正原因是「key 格式混淆 + 專案
    免費額度設定問題」複合，需要分兩步解決

  教訓：API 錯誤訊息不一定指向真正原因。429 不代表「用超過」，可能
  是「key 本身被當成沒配額的身分」。遇到 limit: 0 要先懷疑是身分
  而非用量問題。


● 錯誤 6（Stage 5）：Gemma 模型不支援 Gemini 的 structured output

  我本來想用 gemma-4-31b-it 跑 AI 分析（以為 Gemma 也在 Gemini API
  可用，可以套 responseSchema），結果 API 呼叫成功，但 response 裡
  的文字帶雜訊導致 JSON.parse 失敗，錯誤是：
      Unexpected non-whitespace character after JSON at position 523

  我怎麼辨識的：
  - 看 log 知道是 JSON parse 失敗，而不是 API 本身失敗
  - 推測 Gemma 不支援 responseMimeType: 'application/json' +
    responseSchema（這是 Gemini 獨有功能），Gemma 只會根據 prompt
    做字串生成，輸出常帶 markdown code fence 或說明文字

  調整：改用 gemini-2.5-flash-lite，一次通過，結構化輸出嚴格對應
  schema，不再有雜訊 parse 問題。

  教訓：SDK 支援 ≠ 功能支援。同一個 SDK 底下不同 model family 有
  不同能力邊界，要看 model 文件而不是只看 SDK 說明。


● 錯誤 7（Stage 5 的關鍵教訓）：AI 以為自己要幫忙，結果是傷害

  Gemini 第一次成功 pass checker 後，我檢查實際 sentences，發現 AI
  產出的 text 是這樣：
      「此區間共有 29 個完整資料天數 (factIds: [\"F1\"])」

  AI 自作主張把 factIds 標示塞進 text 裡，同時在 JSON 的 factIds
  欄位也填了 ["F1"]。這段在前端會直接顯示「(factIds: [...])」給使用者
  看到，視覺污染。

  我怎麼辨識的：
  - 我不是看 verdict=pass 就信，而是把每句的 text 列出來肉眼讀
  - 一眼看到英文方括號在中文敘述裡就知道不對

  調整：
  - Prompt 第 7 條明確加上「text 必須是純繁中，不得包含括號、factIds、
    JSON 語法」並附正確 / 錯誤範例
  - Checker 加 R4b 規則，regex /factIds|factId|\[\s*"F\d+"/ 擋掉
    text 中出現的 JSON 語法，作為 prompt 失效時的第二層防線

  教訓：checker 的價值就在這裡 — prompt 一定會有失效的時候（特別是
  新 edge case），deterministic 規則檢查是最後一道不會幻覺的防線。
  我絕對不會讓「看起來 pass 就好」的僥倖上線。


● 錯誤 8：Nuxt SSR 會讓客戶端 devProxy 失效

  第一版主頁面用 onMounted(submit) 讓首次載入自動抓資料。在
  Nuxt SSR 階段這也會執行一次，但 Nitro 內部 fetch 不走 devProxy，
  會打不到 backend 3001。

  我怎麼辨識的：
  - 我預先想過 SSR 與 devProxy 的關係，用 import.meta.client
    guard 住，避免第一次載入就 500
  - 直接 curl http://localhost:3000/ 看 SSR 輸出，確認 <!---->
    佔位符存在（表示 data 為 null 但沒報錯）

  教訓：「能動」有時候是因為第二次就觸發真正的 client fetch 蓋過
  了第一次的失敗；實際要看 server log + 第一次 render 結果。


● 錯誤 9（Stage 4）：Windows bash + curl 把中文 JSON body 編成 Big5

  測試地名模糊匹配時，第一輪 curl 測試結果全部失敗 — 連「台北」「墾丁」
  「玉山」這種白名單內 alias 都被當成「不是台灣地名」，AI 回傳空字串。
  我差點以為 AI 判讀爛，要改用別的 model。

  實際原因：
      curl -d '{"input":"屈尺"}' ...
  在 Git Bash（Windows 10/11）下，shell 把中文字符以系統 locale（Big5,
  codepage 950）編碼才送出，後端收到的 JSON body 是亂碼，AI 看到
  ���B 當然判斷「不是地名」。

  我怎麼辨識的：
  - 第一次 curl 測試時多看了 AI 的 reason 欄位：「使用者輸入的字串
    無法辨識為台灣地名」、「使用者輸入的文字亂碼」
  - AI 說「亂碼」是真正的線索 — 不是 AI 看不懂中文，是 AI 真的收到了
    亂碼

  調整：
      printf '{"input":"%s"}' "$q" > /tmp/body.json
      curl --data-binary @/tmp/body.json ...
  用檔案 POST 強制 UTF-8，結果立刻全通過。

  教訓：跨多層的系統（terminal → HTTP → Node → LLM）每一段都有可能
  做 encoding 轉換，不能假設「我打的字應該是 UTF-8」。AI 的抱怨
  （「亂碼 / 無法辨識」）常常是真正的訊號，要字面理解、不要當成
  AI 能力不足。


─────────────────────────────────────────────
(C).3 我如何調整提問或驗證方式，使結果變得可信？
─────────────────────────────────────────────

1. 先要判斷、再要程式碼
   我把對話切成「範圍取捨 → 逐階段設計 → 寫碼 → 驗證」四段。
   在寫任何 code 之前，我會先要 AI 用文字列出：要做什麼、要放棄
   什麼、為什麼、風險在哪。這層文字產出要我點頭才進下一步。

2. 規則先寫成明確的清單，不要依賴 AI 的預設行為
   比如「缺漏不內插 / 缺漏則 MA=null / AI 禁用趨勢類詞彙 / 白名單
   護欄」，這些題目期待的嚴格行為我都寫進 STAGE?.md 與對話中，
   不讓 AI 自由發揮。

3. 驗證：數字要對拍、schema 要看真實回應
   - 每個端點都先用 curl 打一次、手動看 JSON 結構，再寫 Zod
   - MA5 / MA20 手算後對拍 API 輸出（MA5@04-12 = 30.78，MA20@04-12
     = 26.7，全對）
   - 重要 edge case 專門測：04-13 缺漏日對應 MA=null 的行為

4. 提問語氣：讓 AI push back
   我不只是說「幫我做 X」，而是問「X 有哪些模糊處？你建議砍掉什麼？」
   這讓 AI 能在發現我技術棧過重（Postgres + Cron）時主動提醒我
   砍掉，而不是盲目實作我給的清單。

5. 文件與程式分家
   STAGE1 / STAGE2 / STAGE3 各有一份 markdown，紀錄關鍵決策、踩雷、
   驗證方式、下一步。這些文件不只是交代，更是作答文件 (B)(D)(E)
   的素材來源 — 之後填作答時可以直接引用，不用回憶。


─────────────────────────────────────────────
補充：我「故意不讓 AI 做」的事
─────────────────────────────────────────────

- 不讓 AI 直接從 swagger 推 schema：要求先看真實回應
- 不讓 AI 寫單元測試：3 小時內雙線會雙輸，改手動對拍（見 A.3）
- 不讓 AI 審查 AI（原計畫的雙層架構）：改 deterministic 規則檢查
- 不讓 AI 碰使用者自訂 prompt：攻擊面太大且不可預測
- 不讓 AI 決定範圍：範圍取捨是評分重點，我自己要負責

─────────────────────────────────────────────
結語
─────────────────────────────────────────────

這次使用 AI 的核心原則是：把 AI 當成會幻覺的資深工程師。它
寫 code 速度是我的好幾倍、但文件解讀、邊界條件、真實世界資料型別
這些細節全要我自己用真實證據對拍才敢採用。題目評的是「使用 AI 做
判斷」，我交出的作品核心決策（該砍什麼、該關什麼、哪些規則要寫死）
全部由我把關，AI 只是加速實作的工具。
